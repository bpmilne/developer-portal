Dwolla OAuth 2.0

Overview

Dwolla's API lets you interact with a user's Dwolla account and act on its behalf to send money, request money, and more. To do so, your application first needs to request authorization from users.

Dwolla implements the OAuth 2.0 standard to facilitate this authorization. Similar to Facebook and Twitter's authentication flow, the user is first presented with a permission dialog for your application, at which point the user can either approve the permissions requested, or reject them. Once the user approves, an authorization_code is sent to your application, which will then be exchanged for an OAuth access_token.

Note: Since March 2nd, 2015, newly generated access tokens have a lifetime of 1 hour, and any existing access tokens will begin to expire. Read more about our new OAuth implementation.

User experience

Here's a typical user experience of Dwolla's OAuth + REST API:

  1. Customer initiates a checkout on the merchant site, and clicks on a "Pay with Dwolla" button, which sends the user to the OAuth request authorization URL. If the user doesn't have a Dwolla account, they can create one during the OAuth process. Read more about OAuth account creation.

  2. OAuth permissions dialog pops up.

  3. Customer inputs their PIN. The application then uses this pin along with the user's access_token to Send money to a merchant account.

OAuth flow

First, redirect the user to the permission page:

Generate a URL which describes the permissions your application requires (scope), who the client application is (client_id), and where the user should be redirected to after they grant or deny permissions to your application (redirect_uri). If the user doesn't have a Dwolla account, they can create one during the OAuth process. Read more about OAuth account creation.

require 'dwolla'

# set your application credentials
Dwolla::api_key = @api_key
Dwolla::api_secret = @api_secret

# permissions to request
scope = "send|balance|funding|transactions|accountinfofull"

redirect_uri = "https://www.back2ursite.com/return"
authUrl = Dwolla::OAuth.get_auth_url(redirect_uri, scope)

puts authUrl

# => https://www.dwolla.com/oauth/v2/authenticate?client_id=JCGQXLrlfuOqdUYdTcLz3rBiCZQDRvdWIUPkw%2B%2BGMuGhkem9Bo&response_type=code&redirect_uri=www.back2ursite.com%2Freturn&scope=send|balance|funding|transactions|accountinfofull

require '../lib/dwolla.php';

// URL where user is sent after granting authorization
$redirectUri = 'https://www.back2ursite.com/return'

// set desired permissions
$permissions = array("Send", "Transactions", "Balance", "Request", "Contacts", "AccountInfoFull", "Funding");

$Dwolla = new DwollaRestClient($apiKey, $apiSecret, $redirectUri, $permissions);
$authUrl = $Dwolla->getAuthUrl();

echo $authUrl;

// => https://www.dwolla.com/oauth/v2/authenticate?client_id=JCGQXLrlfuOqdUYdTcLz3rBiCZQDRvdWIUPkw%2B%2BGMuGhkem9Bo&response_type=code&redirect_uri=www.back2ursite.com%2Freturn&scope=send|balance|funding|transactions|accountinfofull

var Dwolla = require('dwolla')('app key here', 'app secret here');

var redirect_uri = 'https://www.back2ursite.com/return';

// permissions to request
var scope = 'send|balance|funding|transactions|accountinfofull';

console.log(authUrl);

// => https://www.dwolla.com/oauth/v2/authenticate?client_id=JCGQXLrlfuOqdUYdTcLz3rBiCZQDRvdWIUPkw%2B%2BGMuGhkem9Bo&response_type=code&redirect_uri=www.back2ursite.com%2Freturn&scope=send|balance|funding|transactions|accountinfofull

from dwolla import DwollaClientApp
Dwolla = DwollaClientApp('app key here', 'app secret here')

# URL where user is sent after granting authorization

# permissions to request

print authUrl

# => https://www.dwolla.com/oauth/v2/authenticate?client_id=JCGQXLrlfuOqdUYdTcLz3rBiCZQDRvdWIUPkw%2B%2BGMuGhkem9Bo&response_type=code&redirect_uri=www.back2ursite.com%2Freturn&scope=send|balance|funding|transactions|accountinfofull

Request Format:

https://www.dwolla.com/oauth/v2/authenticate?client_id={client_id}&response_type=code&redirect_uri={redirect_uri}&scope={scope}

Request Example:

https://www.dwolla.com/oauth/v2/authenticate?client_id=abcdefg&response_type=code&redirect_uri=https%3A%2F%2Fwww.myredirect.com%2Fredirect&scope=send%7Ctransactions

If the user isn't yet logged in, they'll be asked to log in:

Here's what the permissions dialog looks like.

Second, the user will be redirected back to your site.

The user will be taken back to the supplied redirect_uri, along with a temporary code as a GET querystring parameter. Note: This is not an access token, yet. You will need to exchange this temporary authorization code for an actual access_token & refresh_token pair in the next step.

In a successful approval:

{redirect_url}?code={code}

In a failure, or rejection:

{redirect_url}?error=access_denied&error_description=The+user+denied+the+request.

For example, if I set the redirect_uri to https://www.google.com/...

Third, exchange the code for an access token & refresh token pair

In this final step of the OAuth flow, you will exchange the code granted on step 2 for an OAuth access_token and refresh_token pair. In order to do so, you will need to make an HTTP request from your application server, to the Dwolla authentication server.

POST https://www.dwolla.com/oauth/v2/token
Content-Type: application/json

{
  "client_id": "JCGQXLrlfuOqdUYdTcLz3rBiCZQDRvdWIUPkw+pGMuGhkem9Bo",
  "client_secret": "g7QLwvO37aN2HoKx1amekWixa2g0AIuPbD5C/JSLqXIcDOxfTr",
  "code": "wb1JCUPYz6CDrebgBrtTkxx5pKo=",
  "grant_type": "authorization_code",
  "redirect_uri": "https://www.google.com/"
}

code = "put authorization code here"

# we'll need the redirect_uri we used in step 1
redirect_uri = "https://www.google.com/"

token = Dwolla::OAuth.get_token(code, redirect_uri)
puts token

# => '9RoFDRNiH8e1d3pOMPyVGSilGqvlhS9qeFFVgMkx0u0PELhECu'

$code = "authorization code goes here";
$token = $Dwolla->requestToken($code);

Refreshing an Access Token

Handle token expiration

Because access tokens are short lived and expire in 1 hour, if you wish to maintain authorization for a user account for a duration of time longer than 1 hour, you can use the refresh token to generate a new access token and refresh token. There is no limit to the number of times you can refresh a user's access token. Nonetheless, it is important to keep in mind that upon a successful refresh your existing access token and refresh token will be invalidated.

Note: We recommend safely storing a user's refresh token, i.e. in a database, so you can use it in the future to refresh authorization. If you misplace or delete a refresh token, then you will need to repeat the OAuth flow starting from the first step described above.

To refresh your authorization:

POST https://www.dwolla.com/oauth/v2/token
Content-Type: application/json

{
  "client_id": "JCGQXLrlfuOqdUYdTcLz3rBiCZQDRvdWIUPkw+pGMuGhkem9Bo",
  "client_secret": "g7QLwvO37aN2HoKx1amekWi8a2g7AIuPbD5C/JSLqXIcDOxfTr",
  "refresh_token": "vpr47YDbMqlz7EgRjbCqpIZvGWhdJmXjU9zJz8LHy04/ai6lhC",
  "grant_type": "refresh_token"
}

refresh_token = "cIwRQqCIZUW3+FUfCNqK6fCaJ2aSRWK2S1I0mhhIvIXg77cv/X"

puts Dwolla::OAuth.refresh_auth(refresh_token)

$refreshToken = "yURlUzIyYSSP0q+8D9zyJ6wje2BZdFpK4yXG2B69S/+FcUXZVE";
$response = $Dwolla->refreshAuth($refreshToken);
print_r($response)

OAuth Scopes

When authenticating a user into your Dwolla application, you must explicitly specify which access scopes you will require. This helps users better understand what kind of account permissions you require, and which permissions they are granting you application. Multiple scopes may be scoped in your authorization request using the pipe ("|") character (e.g. scope=send|transactions|etc).

Requested scopes must also be enabled on your application settings, otherwise, authenticating will respond with an error stating that your application requested invalid scopes.

Scope Description
AccountInfoFull Detailed information about the authenticated account
Contacts Contacts of the authenticated account
Transactions List and retrieve user's transactions
Balance Dwolla balance of the authenticated account
Send Send money from the authenticated account
Request Initiate a money request, originating from the authenticated account. View, cancel, and fulfill existing requests
Funding View, add, verify Funding sources (i.e. bank accounts) linked to the authenticated account. Initiate deposits from and withdrawals to funding sources
ManageAccount Manage the authenticated user's account settings
