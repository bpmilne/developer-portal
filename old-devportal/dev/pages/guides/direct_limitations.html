Dwolla Direct account limitations

Dwolla Direct accounts have limited functionality compared to full Dwolla accounts. These limitations impact what actions are available via the API.

A key difference is Direct users do not hold a balance within the Dwolla network. All funds received by these accounts are routed straight to the connected funding source, and the Add Funds to Balance and Withdraw Funds from Balance endpoints will not be accessible for Direct users. In addition, the Add Funding Source and Verify Funding Source endpoints will also be unavailable.

Other key items to keep in mind:

  • Direct users can only send money to or receive money from a fully verified account.
  • Any payments sent from a Direct user must be funded by a verified funding source.

Direct users with verified funding source

Direct users with a verified funding source can send money from that verified funding source. Direct users can only send money to recipients with traditional fully verified Dwolla accounts. This lightweight account can send up to $5,000 per week. Direct users do not have a PIN, so the PIN field must be omitted when using the Send API.

The following endpoints are restricted: MassPay, Request from Direct or External and Refund with an error of "User cannot perform this operation".

Direct users with unverified (receive-only) funding source

Direct users with an unverified funding source can receive money to that linked account, but cannot send money from that account. Any API endpoints that involve moving money, such as MassPay, Send, Fulfill Request, Refund are unavailable to these users.

Direct users and PINs

Use the Catalog endpoint to determine whether or not the user needs to provide a PIN in order to use the Send, Create Scheduled, or Fulfill Request endpoints.

The Catalog endpoint returns links to endpoints that can be used by a user. When it's called with a Direct user, some endpoint names in the keys of the _links object will have "WithoutPin" appended (eg. "sendWithoutPin"), which indicates that you will not need to supply a pin to use the endpoint and will not need to ask the user for a PIN.

{
    "Success": true,
    "Message": "Success",
    "Response": {},
    "_links": {
        "self": {
            "href": "http://uat.dwolla.com/oauth/rest/catalog"
        },
        "sendWithoutPin": {
            "href": "http://uat.dwolla.com/oauth/rest/transactions/send"
        },
        "createScheduledWithoutPin": {
            "href": "http://uat.dwolla.com/oauth/rest/transactions/scheduled"
        },
        "fulfillWithoutPin": {
            "href": "http://uat.dwolla.com/oauth/rest/requests/{request_id}/fulfill",
            "templated": true
        }
    }
}

Upgrading Direct accounts

A user may start out with this lightweight Direct account, but may at some point in the future need to use some of these restricted features. In this case, send the user to dwolla.com/direct/upgrade to upgrade their account.
