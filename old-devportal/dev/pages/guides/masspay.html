Sending Batch Payments

Overview

MassPay can be used to disperse payments to many recipients. A common use case is payroll. For instance, WidgetCorp has 950 employees that they need to pay every month. WidgetCorp has a transaction limit of $10,000; and each paycheck is going to be less than that amount, and cannot exceed that amount (though they can easily request a transaction limit increase).

WidgetCorp will need an access token for their account in order to send a MassPay from their account. Read more about OAuth here. Or, WidgetCorp can use the token generator here. The OAuth scopes you'll need are: Send, Funding, and Transactions.

Before we get started, you'll probably want to install one of our client libraries on your server, which is where we'll be adding our code.

First, we'll need to identify the connected bank account from which WidgetCorp will fund their payroll disbursement.

We'll including a nice payment note to Joe and Jane(which is visible from their dwolla.com dashboard), as well as some optional metadata like their employee ID to help us identify this payment within our HR system. Becuase we're sending the payment to their email addresses, we must specify destinationType to be "Email".

Creating a Masspay Job

Now we're ready to create the Job. We'll also assign a custom identifier, "Jan 31 payroll" for this job so that we can recognize the job later.

If the Job was created successfully, Dwolla will respond with the resulting Job's randomly generated Id, Total batch transaction amount, Fees sum of all incurred fees, its Status and an ItemSummary. All jobs have a Status of queued at the time of creation.

Be sure to store Job.Id to retrieve the job in the future and look up its status.

Your MassPay Job will then be queued and processed. As the service processes a Job, each Item is processed one ofter the other, at an average rate of about 0.5 sec. / item. Therefore, you can expect a 1000-item MassPay Job to be completed in about 8 minutes.

<?php
// We need the Masspay class in order to do anything
require '../lib/masspay.php';
$MassPay = new Dwolla\MassPay();

/**
 * Create a MassPay job with two items to
 * the Funding Source of the user associated with the current
 * OAuth token.
 */
$fundsSource = '94c2a1f45853aa43a5c49ffaa7abd759';

$jobInfo = $MassPay->create($fundsSource, $items, [
  'assumeCosts' => true
]);
?>

Checking if a Job is Complete

Now that the Job has been created, WidgetCorp's system will need to periodically check to see if the job has been completed, and if so, if all the payments were successful. There are no webhooks (yet) for MassPay jobs, so we will need to poll Dwolla. We will use the GET MassPay / Job endpoint.

$result = $MassPay->getJob('a3d7cbae-15a8-49ef-87c0-a4210168bbbf');

puts Dwolla::MassPay.getJob('a3d7cbae-15a8-49ef-87c0-a4210168bbbf')

masspay.getjob(jobInfo['Id'])

var jobId = 'a3d7cbae-15a8-49ef-87c0-a4210168bbbf';

dwolla.getMassPayJob(jobId, function(err,result) {
  console.log(result);
});

Example response:

If a job is currently being processed, its Status will be processing and its ItemSummary will indicate how many have been Completed so far, and how many of those were Successful.

When a job is finished, it will have a Status of complete.

Reconciliation

Once the job is complete, we'll want to know the transaction IDs for each of the resulting payments so that we can let our employees know that their payment is on its way. (Joe is infamous for being very impatient about receiving his paycheck). To do this, we'll use the MassPay / Items endpoint to fetch the Items of our Job.

Each Item has a status, a globally unique ItemId, a resulting TransactionId, and any Error encountered with attempting to process the Item.

$jobItems = $MassPay->getJobItems($info['Id']);

puts Dwolla::MassPay.getItems('a3d7cbae-15a8-49ef-87c0-a4210168bbbf')

jobItems = masspay.getjobitems(jobInfo['Id'])

var jobId = 'a3d7cbae-15a8-49ef-87c0-a4210168bbbf';

dwolla.getMassPayJobItems(jobId, function(err,jobItems) {
  console.log(jobItems);
});

Checking Resulting Transactions

An Item is distinct from the transaction which it creates. When an Item has been processed and has a status of "Success", that merely means that a transaction was created. The transaction itself may or may not have been cleared successfully in the recipients account. It is important for WidgetCorp to know whether or not that transaction was cleared as successful, for it could have failed because of an ACH error, such as a lack of sufficient funds in their bank account. For each Item, we will fetch the resulting Transaction resource using the the Transaction / Id endpoint.

Example: Checking the transaction status of Joe's payment

// We need the Transactions class in order to do anything
require '../lib/transactions.php';
$Transactions = new Dwolla\Transactions();

$Transactions->info('812270'));

puts Dwolla::Transactions.get(812270)

from dwolla import transactions, constants

# Configure the library (change these)
constants.sandbox=True

constants.client_id = "Client Id goes here"
constants.client_secret = "Client secret goes here"
constants.access_token = "access token goes here"
constants.pin = 1234

# Get info for transaction ID '812270'.

transactions.info('812270')

/**
 * Retrieve information about transaction ID '812270'
 * from the user with the authorized OAuth token.
 */

Dwolla.transactionById('812270', function(err, data) {
  if (err) { console.log(err); }
  console.log(data);
});
