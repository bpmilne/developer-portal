Dwolla White Label: Transfer funds to your customers

Dwolla’s White Label solution allows you to integrate a powerful ACH payments platform within your application or online experience. You customize the payments experience end-to-end, including customer onboarding, payment notifications, and how your business name is displayed on your customer’s bank statement. In addition to providing an easy to integrate API, Dwolla assumes the role of payments originator. As a part of this function, Dwolla provides its risk and fraud management infrastructure and helps to manages the compliance requirements associated with processing ACH payments.

In this guide, we’ll cover the key points of White Label funds transfer: (1) the creation of a customer, (2) the association of a funding source (bank or credit union account) with the customer and (3) the transfer of funds to the customer’s linked account. The ability to receive funds from a customer or transfer funds between customers requires varying levels of verification and will be covered in subsequent guides.

Quick overview

  1. Use our test application or create a sandbox account.
  2. Record your customer’s acceptance of the Dwolla Terms of Service and create a customer object.
  3. Attach a funding source to the customer.
  4. Subscribe to webhooks to send required communications to the customer.
  5. Transfer funds from your account to the customer.

Get set up in the sandbox

To test the API requests made in this guide, you can either use our sample credentials or generate your own sandbox credentials.

Or, set up your own sandbox credentials

  1. Create a master sandbox account. Step-by-step instructions »
  2. Create an application, generate API credentials, and contact Dwolla at api@dwolla.com to enable the ManageCustomers scope. Not sure how to get API credentials? Step-by-step instructions »
  3. Create an access token. Use your application’s key and secret from the previous step. Select the scopes needed for your application. For example, select: Send, Funding, Transactions, and ManageCustomers. With these scopes, you can complete the OAuth flow, which issues you an access and refresh token pair that contains the proper scopes for creating and managing customers.

Create a customer

Create a customer for each user you’d like to transfer funds to. At a minimum, provide the user’s full name and email address to create the customer.

Request:

POST https://api-uat.dwolla.com/customers
Content-Type: application/vnd.dwolla.v1.hal+json
Accept: application/vnd.dwolla.v1.hal+json
Authorization: Bearer 0Sn0W6kzNicvoWhDbQcVSKLRUpGjIdlPSEYyrHqrDDoRnQwE7Q
{
  "firstName": "John",
  "lastName": "Doe",
  "email": "johndoe@email.com",
  "ipAddress": "127.0.0.1"
}

Response: 

HTTP/1.1 201 Created
Location: https://api-uat.dwolla.com/customers/C7F300C0-F1EF-4151-9BBE-005005AC3747

When the customer is created, you’ll receive the customer object URL in the location header.

Important: We recommend that you provide the IP address of the end-user accessing your application as the ipAddress parameter. This enhances Dwolla’s ability to detect fraud. Sending random, hardcoded, or incorrect information in the ipAddress field will cause delays or throttling of requests.

Attach a funding source to the customer

The next step is to attach a bank or credit union account to the customer by providing the bank account’s routing number, account number, and account type. Funds transferred to this customer will be automatically swept into the funding source.

Request:

POST https://api.dwolla.com/customers/C7F300C0-F1EF-4151-9BBE-005005AC3747/funding-sources
Content-Type: application/vnd.dwolla.v1.hal+json
Accept: application/vnd.dwolla.v1.hal+json
Authorization: Bearer 0Sn0W6kzNicvoWhDbQcVSKLRUpGjIdlPSEYyrHqrDDoRnQwE7Q
{
    "routingNumber": "073000228",
    "accountNumber": "123456789",
    "type": "checking",
    "name": "John Doe - Checking"
}

Response:

HTTP/1.1 201 Created
Location: https://api-uat.dwolla.com/funding-sources/375c6781-2a17-476c-84f7-db7d2f6ffb31

The bank or credit union account details are securely stored with Dwolla. You’ll use the funding source ID to reference the account in all future requests. This ID is found in the location header of the response, e.g. 375c6781-2a17-476c-84f7-db7d2f6ffb31.

Create a webhook subscription

Any customer-related activity is recorded as an event. A webhook subscription allows your application to be notified when events occur and provide timely notifications to your customers.

In order to create a webhook subscription, you’ll need a client access token. This is generated using the client credentials grant.

Request:

POST https://www.dwolla.com/oauth/v2/token
Content-Type: application/json
{
  "client_id": "EQ5OSON1MwpnGoLqlzcoz5ugmEYwjN4atqCJ1jaYhtaiPEb4H7",
  "client_secret": "nrJzCyJTpGcqD5gYH4hkUwk6H5purksHDg4GUm2crDZYMYctTF",
  "grant_type": "client_credentials"
}

Response:

HTTP/1.1 200 OK
{
  "access_token": "SpXcZAtVhCdglXBK3qoVuEenPad5vB8Tm6NhzH6NlhXMq0kpUI",
  "token_type": "bearer",
  "expires_in": 1578000000,
  "scope": "accountinfofull|contacts|transactions|balance|send|request|funding|manageaccount|scheduled|managecustomers"
}

Next, you'll create a webhook subscription by calling the webhook-subscriptions endpoint with the callback URL where you’d like to receive the HTTP requests from Dwolla and a secret string of your choosing. The secret allows you to validate that the webhook originated from Dwolla and not a malicious third party. Each webhook contains a signature which is a HMAC-SHA1 hash of the request body, using the secret as the hash key.

Request:

POST https://api-uat.dwolla.com/webhook-subscriptions
Accept: application/vnd.dwolla.v1.hal+json
Content-Type: application/vnd.dwolla.v1.hal+json
Authorization: Bearer SpXcZAtVhCdglXBK3qoVuEenPad5vB8Tm6NhzH6NlhXMq0kpUI
{
  "url": "http://myapp.com/dwolla/webhooks",
  "secret": "my secret"
}

Response:

HTTP/1.1 201 Created
Location: https://api-uat.dwolla.com/webhook-subscriptions/dc9b4778-362e-4778-b558-15ff2c684584

Responding to Webhooks

When your application/server receives a webhook from Dwolla, it should respond with a 200/OK within 20 seconds. If a webhook is not received or you don’t respond properly, Dwolla will re-attempt delivery 8 times over the course of the following 72 hours. If your application fails to receive a webhook, it can retrieve all webhooks ever created.

A webhook looks like:

{
  "_links": {
    "self": {
      "href": "https://api-uat.dwolla.com/events/a58be29f-ce03-4b42-9d6c-48ad2b4093ee"
    },
    "resource": {
      "href": "https://api-uat.dwolla.com/transfers/3B192430-8D47-E511-80DA-0AA34A9B2388"
    }
  },
  "id": "a58be29f-ce03-4b42-9d6c-48ad2b4093ee",
  "timestamp": "2015-08-20T22:45:44.000Z",
  "accountId": "ad5f2162-404a-4c4c-994e-6ab6c3a13254",
  "topic": "transfer_created",
  "resourceId": "3B192430-8D47-E511-80DA-0AA34A9B2388"
}

Testing webhooks

RequestBin allows you to create a URL that will collect any requests made to it. You can then inspect your requests and see the values returned. The URL provided is temporary and can only be used for 20 requests or for 48 hours (whichever comes first).

After you create a new RequestBin, you’ll be given a Bin URL to set as your endpoint. Set the Bin URL as the URL to POST to in the features page of your application settings. After you have made a transfer request, refresh your page on RequestBin and you should see the webhook notification from Dwolla. Dwolla includes an X-Dwolla-Signature header, which is an HMAC hash of the request body salted using the consumer application secret.

Create a transfer from your account to a customer

Now that you’ve created a customer, associated a funding source with it, and set up a webhook subscription, you can initiate your first transfer. The transfer requires the following information:

  1. The funding source to pull the funds from (your linked bank account)
  2. The customer to push the funds to

Dwolla uses URLs to represent relations between resources. Therefore, you’ll need to provide the full URL of the funding source and customer.

1. Fetch a list of available funding sources

Use the List Funding Sources (Account) endpoint to fetch a list of your account’s funding sources. You first need to fetch the root resource to determine the URL to get the account’s funding source list from.

Request:

GET https://api-uat.dwolla.com/
Accept: application/vnd.dwolla.v1.hal+json
Authorization: Bearer 0Sn0W6kzNicvoWhDbQcVSKLRUpGjIdlPSEYyrHqrDDoRnQwE7Q

Response:

{
  "_links": {
    "account": {
      "href": "https://api-uat.dwolla.com/accounts/dcbb698d-bee7-4f79-8576-e4301bdc57fc"
    },
    "funding-sources": {
      "href": "https://api-uat.dwolla.com/accounts/dcbb698d-bee7-4f79-8576-e4301bdc57fc/funding-sources"
    }
  }
}

2. Create a transfer.

Create a transfer by specifying the account’s funding source as the source and the customer as the destination.

Request:

POST https://api-uat.dwolla.com/transfers
Accept: application/vnd.dwolla.v1.hal+json
Content-Type: application/vnd.dwolla.v1.hal+json
Authorization: Bearer 0Sn0W6kzNicvoWhDbQcVSKLRUpGjIdlPSEYyrHqrDDoRnQwE7Q
{
    "_links": {
        "destination": {
            "href": "https://api-uat.dwolla.com/customers/C7F300C0-F1EF-4151-9BBE-005005AC3747"
        },
        "source": {
            "href": "https://api-uat.dwolla.com/funding-sources/297460a0-101b-498c-8184-2eb33ff22d34"
        }
    },
    "amount": {
        "currency": "USD",
        "value": "225.00"
    },
    "metadata": {
        "customerId": "8675309",
        "notes": "For work completed on Sept. 1, 2015"
    }
}

Response:

HTTP/1.1 201 Created
Location: https://api-uat.dwolla.com/transfers/d76265cd-0951-e511-80da-0aa34a9b2388

Notify your customer about the transfer

The receipt of a webhook indicates a change in activity for the customer or transfer and can be used within your business workflow. For example, when your application receives the webhook for a created transfer and subsequent information which indicates the outcome of the transfer, you must trigger an email to your customer with the appropriate status update.

Note: Timely and reliable email communication about transfers is a requirement for integrating with Dwolla’s White Label solution.

Refresh the OAuth account access token

Since OAuth access tokens expire after 1 hour, you need to refresh the access token to maintain access to your account and customers. Tokens can be refreshed as often as needed.

Request:

  "refresh_token": "vpr47YDbMqlz7EgRjbCqpIZvGWhdJmXjU9zJz8LHy04/ai6lhC",
  "grant_type": "refresh_token"
