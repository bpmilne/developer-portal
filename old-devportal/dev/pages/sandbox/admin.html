The Sandbox Admin interface lets you batch create test users, manipulate transaction states, and manage your test users in the Sandbox environment. It requires a sandbox account to use. Read about the Sandbox environment »

Go to Sandbox Admin »

Creating Users

Users are created by uploading a CSV containing details which you'd like to create each account with. Each row describes a single account's Email address, Password, PIN, First name, Last name, Dwolla balance, Status ("verified" or "unverified"), in that order, delimited with a semicolon.

For example, uploading the following CSV would create 2 accounts, one of which would be a fully verified account for John Doe, with an initial balance of $5000 and PIN of 1234:

johndoe@gmail.com;Password1!;1234;John;Doe;5000;verified
janedoe@gmail.com;Password1!;1234;Jane;Doeman;100;unverified

Setting Transaction Statuses

Transaction Behavior in the Sandbox

Unlike Balance, Credit, or FiSync sourced transactions, which are processed instantaneously, Bank-sourced transactions exist in the Pending state for several business days until they are Processed, Failed, Cancelled, or Reclaimed. For instance, it may take between 3 to 5 business days before a Pending transaction becomes Processed (which means it has cleared successfully). For a transaction to become Failed in our production environment, Dwolla must receive notice from the ACH network (in the form of an ACH return code), a process which requires at least 1 day. Therefore, a failed payment due to insufficient funds, can be in the Pending state for 3 business days until Dwolla receives notice that the transaction failed due to insufficient funds and its state is updated to Failed.

The Sandbox environment does not replicate any ACH processes, so a Pending transaction will not clear or fail automatically after a few business days as it would in production. It will simply remain in the Pending state indefinitely.

Cancelled and Reclaimed

As in the production environment, transactions can be cancelled by either party from either the sender's or the receiver's Dwolla account dashboard. You may also cancel payments directly from the Sandbox Admin interface.

There are two scenarios in which a transaction can become Reclaimed:

  1. a payment is sent to a non-Dwolla user's email address or phone number and has not been claimed by the sender's Reclaim period, which is by default 30 days.
  2. a processed payment is reversed by Dwolla as a result of a dispute.

Manipulation

The Sandbox Admin interface allows you to immediately trigger any of the four possible transaction outcomes for a given transaction: Processed, Failed, Cancelled, Reclaimed. Since the Sandbox environment is an exact copy of our production environment, the typical events that would follow in our production environment will occur:

  1. Relevant email is sent.
  2. Webhook notification is sent.

Simulating Bank Deposits and Withdrawals

Bank deposits to and bank withdrawals from a Dwolla account are deposit and withdrawal type transactions, respectively. As such, their transaction outcome can be triggered from the Sandbox Admin interface by selecting Process or Cancel.

To deposit money into a Dwolla account from the test bank account, Midwestern Bank XXX6789, initiate the deposit from the account dashboard and Process the resulting deposit transaction from the Sandbox Admin interface. Withdrawals can be completed in the same way.

Account Management

The Sandbox Admin interface allows you to deactivate any accounts you've created. This is useful for understanding the user experience for a user whose account is disabled, and for a user attempting to send money to a user whose account is deactivated.
